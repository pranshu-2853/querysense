MASTER INTERVIEW TALKING POINTS
(Prepare these answers before any interview)
===============================================================

Q: Why JSQLParser instead of regex for SQL validation?
A: SQL is not a regular language. Regex cannot parse nested subqueries,
   quoted identifiers containing SQL keywords, or comment syntax variants.
   JSQLParser builds a full AST — statement type is a Java class, not a
   string match — making it immune to case variations, whitespace injection,
   and comment injection that defeat any regex approach.

Q: Why two PostgreSQL databases instead of one?
A: Defense in depth. Even if JSQLParser were bypassed (a bug, a library
   vulnerability, a compromised application binary), the querysense_readonly
   role's PostgreSQL-level privileges make any destructive SQL impossible.
   The database cannot be compelled to execute a query the role lacks
   privilege to run.

Q: Why pgvector instead of Pinecone or Qdrant?
A: The access group filter on the semantic cache lookup is a SQL JOIN
   predicate: WHERE qj.group_id = $2. In a dedicated vector database,
   this requires metadata filtering — less expressive and requiring
   denormalized metadata. In pgvector, it's a single JOIN clause.
   For relational-vector join requirements at this scale, pgvector is
   architecturally superior. No second data store, no credentials,
   no cross-system join complexity.

Q: How do you know your system is working correctly?
A: The regression evaluation pipeline. I maintain a golden dataset of
   natural language question → expected SQL pairs. An evaluation endpoint
   runs the full pipeline against every golden query and reports accuracy.
   This runs in CI — if accuracy drops below 80%, the build fails. Same
   concept as test coverage thresholds, applied to AI output quality.

Q: What happens when the LLM generates wrong SQL?
A: Three things happen in sequence. First, JSQLParser validates the AST
   — syntax errors, non-SELECT statements, non-whitelisted table references
   are all caught before execution. Second, if validation fails, a correction
   prompt is built that includes the original question, the rejected SQL, the
   specific failure reason, and the list of allowed tables — then Stage 4
   is retried. Third, if execution fails (DB error, timeout), the DB error
   message is fed back into another correction prompt and retried. Maximum
   2 retries total.

Q: Why Spring AI 1.0.x and not LangChain4j?
A: LangChain4j is an abstraction over LLM orchestration patterns. For a
   project whose interview value is demonstrating understanding of those
   patterns, using a framework that hides them is counterproductive. Building
   the pipeline stages explicitly in Spring Boot means every design decision
   — structured output, few-shot retrieval, correction loops, cost tracking
   — is visible and attributable to deliberate engineering choices.

Q: How do you handle temporal expressions like "last quarter"?
A: Deterministically, before the LLM call. A TemporalExtractor maps common
   phrases to concrete date ranges using java.time at request time.
   "Last quarter" resolved at 2:00 AM on December 31 is different from
   at 11:59 PM on the same day if you let the LLM handle it. Resolved
   dates are injected into the prompt as explicit context: "Note: 'last
   quarter' refers to 2025-10-01 through 2025-12-31." This eliminates
   an entire class of LLM date-handling errors.

Q: What is your cache invalidation strategy?
A: Two levels. Exact hash cache (SHA-256 of normalized question text) in
   Redis with 1-hour TTL. Semantic cache (pgvector cosine similarity) also
   with 1-hour TTL. TTL is chosen to balance cost savings against data
   staleness — analytics data changes as business operations proceed.
   When schema sync runs (a column renamed, a table added), Redis schema:version
   is incremented and included in cache keys, invalidating stale entries
   without a full FLUSHDB.
